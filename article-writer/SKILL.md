# 文章写手 - Article Writer

从选题卡或想法生成公众号母稿，遵循黄叔文风，自动关联素材库。

## 触发场景

- topic-partner 对话结束后用户说"直接写吧"
- "写文章：xxx"（指定选题名称）
- "帮我把这个写成公众号文章"
- "出一篇文章"
- `/article-writer`
- `/写文章`

## 不触发场景

- 只想聊选题不想写（用 topic-partner）
- 已有 Markdown 只需排版（用 wechat-formatter）
- 已有文章要发布（用 article-publisher）

## 输入来源

三种输入，优先级从高到低：

1. **选题卡**：从 `01-内容生产/选题管理/01-待深化选题/` 读取，包含核心判断、内容骨架、可复用素材
2. **对话上下文**：从 topic-partner 对话中直接衔接过来
3. **用户直接给的想法**：一段文字、一个观点、一篇参考内容

## 工作流

### Step 1: 准备素材

#### 1.1 读取输入
- 如果有选题卡，读取全部内容
- 如果是直接想法，先快速提炼核心判断和内容方向

#### 1.2 检索知识库和素材库（静默执行）
- `04-知识编译/wiki/concepts/` — 已编译的概念条目（优先引用）
- `04-知识编译/wiki/methods/` — 已编译的方法论
- `04-知识编译/wiki/indexes/index.md` — 全局索引，快速定位相关知识
- `01-内容生产/内容素材库/核心概念库/` — 可复用的理论框架
- `01-内容生产/内容素材库/金句库/` — 高质量表达
- `01-内容生产/内容素材库/案例库/` — 相关案例
- `01-内容生产/内容素材库/爆款文稿库/` — 参考已验证的内容结构
- `00-核心概念词典.md` — 确保术语一致性
- `01-内容生产/文风指南.md` — 写作风格规范

#### 1.3 调取个人认知（静默执行，让文章更像黄叔）
- `00-我/thinking_patterns/` — 检索与选题相关的真实决策、思考、经历，优先用作案例或论据
- `00-我/profile/mbti_profile.md` — 确保文章语气符合 INTP 思维方式（逻辑框架+多可能性+直接）
- `00-我/profile/core_values.md` — 涉及观点输出时校准价值观一致性
- `00-我/profile/gallup_strengths.md` — 内容方向偏向搜集→学习→前瞻→行动的优势链
- `00-我/goals/active_goals.md` — 确保内容方向服务当前核心目标

#### 1.4 时机与独特性校验（必须输出给用户确认）

搜索本周相关热点，向用户展示：

```
## 动笔前确认

### 时机
- 本周相关热点：[搜索结果]
- 挂载角度建议：[怎么把想法接到热点上]
- 时机判断：✅ 现在合适 / ⚠️ 需要调整切入角度 / ❌ 建议等更好时机

### 独特性
这篇文章的不可替代性来自：
- [ ] 一手数据/实验
- [ ] 独特职业视角
- [x] 具体判断项（从输入中判断）
- [ ] 反共识判断

独特性来源：[一句话说清楚]

---
确认没问题我就开始写，或者告诉我调整方向。
```

**用户确认后才能继续。**

### Step 2: 写 Markdown 母稿

#### 2.1 文风要求 — 黄叔风格

1. **开头不装**
   - ❌ "在当今 AI 时代，工具的迭代速度..."
   - ✅ "前两天有个朋友问我，为什么他的团队买了一堆 AI 工具，结果没人用。"

2. **说人话**
   - ❌ "此外，我们需要关注用户的认知负担问题"
   - ✅ "还有个更要命的问题：用户根本不想学。"

3. **判断要硬 + 半数皱眉测试**
   - 核心观点必须让一半人点头、另一半人皱眉
   - ❌ "或许我们可以考虑简化功能"
   - ✅ "功能多不是优势，是负担。"

4. **数据要真**
   - 只用真实案例和数据，没有就说"我观察到..."

5. **章节过渡用悬念，不用总结**
   - ❌ "综上所述，这个方法值得尝试。"
   - ✅ "这个结论看起来很显然——但有个细节我差点忽略了。"

6. **金句要具象，不要抽象**
   - ❌ "功能多不是优势，是负担"
   - ✅ "买了 Notion 全套，半年后打开率最高的还是 Excel"

7. **逻辑链要通**
   - H2 标题 + 每节核心句连读，不看正文也能读懂结论

8. **收尾写转变，不写总结**
   - ❌ "希望本文对您有所启发"
   - ✅ "下次做产品，先问自己：用户需要选择，还是需要答案？"

9. **用名人故事/案例增强说服力**
   - 每篇文章至少引用 1-2 个名人故事、商业案例或经典类比来强化核心观点
   - 名人故事的作用：让抽象判断变得具象，让读者通过"别人的故事"理解"你的道理"
   - 选择标准：
     - ✅ 和文章核心观点有**结构性类比**关系（如"龙虾 = WordPress"）
     - ✅ 读者大概率知道这个人/公司（降低理解成本）
     - ✅ 故事本身有**反直觉**或**转折感**
     - ❌ 不要硬凑名人名言，不要"马云说过..."这种鸡汤式引用
   - 好的用法示例：
     - 用 WordPress 的发展史类比龙虾的定位（结构性类比）
     - 用 Basecamp 拒绝做大公司的故事呼应"拒绝高客单"（价值观共鸣）
     - 用 Notion 早期坚持 ToC 低价的决策映射自己的选择（同类对标）
   - 位置建议：通常放在核心判断之前或之后，作为"证据"或"共鸣放大器"

#### 2.2 去 AI 味规则（自动执行）

**替换规则**：
- "在当今社会" → "最近发现个有意思的事"
- "随着科技的发展" → "前几天有个朋友问我"
- "此外" → "还有啊" / "对了"
- "然而" → "但问题是"
- "因此" → "所以啊"
- "综上所述" → "说白了"
- "希望本文对您有所帮助" → "希望能帮到你，有问题随时问"

**绝对禁用词**：毋庸置疑、在当今时代、事实上（→其实）、笔者认为（→我觉得）、赋能/助力/抓手

**口语化**：用"你"直接对话，每 300 字至少 1 处口语标记（"说白了"、"讲道理"、"老实说"）

#### 2.4 配图占位符（必须包含，不可省略）

格式：`![图注](SLUG_imgN_keyword.png)`
- 每 500-800 字一张，2000 字以内 2-3 张，2000-3000 字 3-4 张
- SLUG 从标题提取 8-12 字中文关键词
- **图注必须是描述性的中文短句**，发布时会作为图片下方说明文字
- 占位符位置建议：每个 PART/章节结束后、关键概念阐述后、流程说明后

⚠️ **重要**：如果母稿没有图片占位符，article-publisher 的 pipeline 会跳过配图步骤，导致发布的文章没有任何插图。这是过去实际踩过的坑——务必在写作时就规划好配图位置。

### Step 3: 对抗式内容审核（核心质量关卡，不可跳过）

> 直接让 AI 写，给你的是"能用"；对抗式给你的是"经得起质疑"。差距很大。

写完初稿后，启动三角色对抗循环。**角色必须分离**——写的人不能自己审自己，必须切换到完全不同的立场。

#### 三个角色

| 角色 | 职责 | 立场 |
|------|------|------|
| **笔杆子** | 写稿、改稿 | 创作者：为每一句话辩护 |
| **参谋** | 找问题、攻击 | 批评者：只挑刺，不帮写，不留情面 |
| **裁判** | 综合评分、判生死 | 决策者：看完草稿也看完骂稿，给最终裁决 |

#### 3.1 参谋攻击（五维度逐一攻击）

参谋不帮你写，只做一件事——**找问题**。按以下五个维度逐一攻击，每个维度必须给出具体问题和改进要求：

| # | 攻击维度 | 攻击方向 | 具体审问 |
|---|---------|---------|---------|
| 1 | **标题与开场** | 能让人点开吗？ | 标题有没有信息差/悬念/反直觉？前 3 句有没有异常信号？放在信息流里会被划走吗？开场是在"说背景"还是在"制造好奇"？ |
| 2 | **结构与逻辑** | 站得住脚吗？ | H2 标题连读自洽吗？论证链有没有断裂或跳跃？章节过渡是总结句（❌）还是悬念/问题（✅）？有没有哪个章节删掉不影响全文？ |
| 3 | **数据与可信度** | 经得起追问吗？ | 哪里是空话？数据来源可查吗？没数据的判断有没有用"我观察到"？有没有编造案例？核心判断通过"半数皱眉测试"了吗？ |
| 4 | **读者价值** | 看完有收获吗？ | 读者读完能做什么？会不会中途弃读？有没有"so what"时刻？金句是具象的还是抽象的？能说出"这篇只有我能写，因为___"吗？ |
| 5 | **篇幅与节奏** | 比例合理吗？ | 哪里写多了需要砍？哪里写少了需要补？口语感够不够（每 300 字至少 1 处口语标记）？有没有 AI 味禁用词？节奏有没有拖沓？ |

**参谋语气要求**：直接、尖锐、不客气。不说"建议考虑"，说"这里不行，必须改"。

参谋输出格式：
```
## 参谋攻击报告（第 N 轮）

### 1. 标题与开场 — [通过/不通过]
- 问题：[具体指出哪里不行]
- 要求：[必须怎么改]

### 2. 结构与逻辑 — [通过/不通过]
- 问题：...
- 要求：...

### 3. 数据与可信度 — [通过/不通过]
...

### 4. 读者价值 — [通过/不通过]
...

### 5. 篇幅与节奏 — [通过/不通过]
...

**总结**：[一句话概括最大的问题是什么]
```

#### 3.2 笔杆子修改

根据参谋攻击报告，笔杆子逐一回应并修改文章：
- 每个攻击点必须要么**修改**，要么给出**不改的理由**（理由必须站得住脚）
- 修改后简要标注改了什么

#### 3.3 裁判评分

裁判看完修改后的文章和参谋的攻击报告，给出 0-10 分的综合评分：

| 分数 | 判定 | 动作 |
|------|------|------|
| 9-10 | 优秀 | 通过，输出最终版本 |
| 8 | 合格 | 通过，输出最终版本 |
| 6-7 | 有短板 | **打回重写**，重新进入 3.1 |
| ≤5 | 不合格 | **打回重写**，参谋重点攻击上轮最弱的维度 |

裁判输出格式：
```
## 裁判评分（第 N 轮）

**评分：X/10**

| 维度 | 得分 | 一句话评价 |
|------|------|-----------|
| 标题与开场 | X/10 | ... |
| 结构与逻辑 | X/10 | ... |
| 数据与可信度 | X/10 | ... |
| 读者价值 | X/10 | ... |
| 篇幅与节奏 | X/10 | ... |

**判定：[通过/打回]**
**理由：[为什么通过或为什么打回，指出最关键的问题]**
```

#### 3.4 循环机制

```
笔杆子写初稿 → 参谋攻击 → 笔杆子修改 → 裁判评分
                  ↑                              ↓
                  └──── 低于 8 分，打回重写 ←──────┘
```

- **最多循环 3 轮**
- 每轮：参谋攻击 → 笔杆子修改 → 裁判评分
- **≥ 8 分即通过**，输出最终版本
- 3 轮后仍未达标：向用户展示当前版本 + 各轮攻击记录 + 瓶颈分析，请用户决定是发布还是继续调整

#### 用户看到的 vs 看不到的

用户**看不到**的：被骂死的初稿、标题改了三遍的记录、被参谋批得体无完肤的版本、裁判说 5 分重写的记录。

用户**看到的**：经过反复打磨的最终版本 + 对抗过程摘要（跑了几轮、每轮核心改动）。

#### 为什么用对抗式替代自检

| | 旧方案（十维度自检 + 专家审阅） | 新方案（对抗式内容审核） |
|---|---|---|
| 本质 | 自己检查自己 | 角色分离，互相对抗 |
| 迭代 | 一次性检查 | 循环打磨 2-3 轮 |
| 力度 | 打勾式检查，容易放水 | 参谋不留情面，专门找茬 |
| 结果 | "能用" | "经得起质疑" |

## 与其他 skill 的衔接

### 上游：topic-partner
- topic-partner 输出选题卡后，用户说"直接写吧" → 本 skill 接手
- 读取选题卡的核心判断、内容骨架、可复用素材作为写作输入

### 下游：article-publisher
- 本 skill 输出母稿后，用户说"发布" → article-publisher 接手
- 负责生图、HTML 排版、封面、发布到草稿箱

### 下游：平台派生（内置在本 skill）
- 用户说「出小红书版」「出视频脚本」「出播客版」
- 读取同一文件夹内的母稿，按平台特征适配
- 保存到同一选题文件夹，更新 _meta.md

## 平台派生规则

### 小红书版
- 从母稿提炼 3-5 个核心要点
- 标题：加 emoji、数字、疑问句，15 字以内
- 正文：口语化、分段短、每段 1-2 句
- 适合分页图文的结构

### 抖音/视频号脚本
- 前 5 秒钩子（反直觉/痛点/数字冲击）
- 口播脚本，控制在 60 秒以内
- 标注画面切换点和字幕重点
- 结尾引导互动（"你觉得呢？评论区聊聊"）

### 小宇宙播客提纲
- 展开式结构，比文章更散更深
- 对话语气，像在和朋友聊天
- 标注重点段落和可展开的故事
- 预估时长

## 注意事项

1. **先确认再写**——时机与独特性校验必须等用户确认
2. **对抗式审核不可跳过**——写完必须经过参谋攻击 + 裁判评分，低于 8 分必须打回重写
3. **素材库优先**——有可复用的素材必须用，不要从零开始编
4. **不编造数据**——没有数据就说"我观察到..."
5. **母稿制**——公众号版就是母稿，其他平台版本都从它派生

## 自我记录规则

每次完成主任务后，必须在 `Learnings.md` 里追加一条记录：

```
## 日期：YYYY-MM-DD
- 任务类型：xxx
- 输入摘要：用户这次给了什么（一两句话概括）
- 执行结果：成功 / 部分成功 / 失败
- What Worked：哪些做法效果好
- What Failed：哪些做法翻车了（如果有）
- 特殊情况：这次遇到的边缘场景
```

积累到一定数量后（比如用满一两周的高频场景），读取 `Learnings.md`，分析是否有可以固化到 `SKILL.md` 的规律。如果有，更新 `SKILL.md` 的相应章节，并标注"从 Learnings.md 提炼于 YYYY-MM-DD"。
